## Notes for continuity

- **Concurrency is now two mechanisms, not one — keep them separate.** The advisory lock (`locked_by`/`locked_at`, unchanged from before this session) still means "the last session to hold this either hasn't closed it yet, or didn't close it cleanly." **Read-Only** (`WorkfileState.read_only`, new this session) is a separate opt-in session mode that never claims that lock. Don't conflate "locked" with "read-only" when writing this up — a workfile can be locked and opened normally (taking on the risk), or unlocked and opened read-only (no risk, no reason given other than the user's own choice).
- **Read-Only scope is shallow by deliberate decision, not oversight.** Only Save is disabled. Every other tab, including the Outputs tab's batch runs, behaves exactly as in a normal session. Edits made in a read-only session that aren't rescued via Save As are genuinely lost, same as Word/Excel's read-only model. Don't propose extending Read-Only enforcement to individual widgets without raising it fresh — it was scoped down explicitly after discussion.
- **The "different folder" rule on Read-Only's Save As is about `outputs/` collision, not overwrite protection.** Exact-path overwrite was already guarded before this session. Two workfiles sharing a folder would share `outputs/pptx`/`outputs/pdf`, mixing reports from what should be independent workfiles. Get this reasoning right in the docs — it's not the same justification as ordinary Save As's overwrite check, and ordinary (non-read-only) Save As still permits same-folder-different-name today, unchanged, by earlier decision.
- **`close_workfile` now branches on `state.read_only` before touching the lock at all.** Any future code that calls `clear_lock` or `write_lock` directly (bypassing `close_workfile`/`open_workfile`'s existing entry points) needs the same guard applied by hand — there's no central enforcement beyond those two functions.
- Two things were discussed and explicitly deferred, not forgotten: a `beforeunload` browser warning on tab close (addresses the separate "forgot to save" problem, not this session's scope, still worth doing), and any autosave/checkpoint mechanism (raised once earlier, not picked up again this session). Worth raising as fresh topics rather than assuming either was rejected.
- The login-screen lock-warning path (`show_login`'s `lock_info` handling, tied to `pending_workfile_path`) was left untouched — it's dead code, unreachable because nothing in the app currently sets `pending_workfile_path` (it's vestigial groundwork for the not-built `.cgw` file-association feature).
